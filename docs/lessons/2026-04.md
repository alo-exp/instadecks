### stack:pptxgenjs: pptxgenjs version pinning + ShapeType enum gotchas + timestamp non-determinism

**When this applies:** You depend on pptxgenjs (or any deck-generation library) for visually calibrated output, and downstream consumers (annotation overlays, visual diff) assume exact geometry.

Pin pptxgenjs to an exact version (no caret); minor releases shift glyph metrics, default theme colors, and shape-builder defaults enough to break pixel-aligned overlays. Always use the `ShapeType` enum (`pres.shapes.RECTANGLE`) rather than string literals — string literals silently degrade to default rectangles in some versions. Generated PPTX files contain creation timestamps and absolute-path image `descr` attributes, so byte-identical visual regression is impossible; normalize the XML (strip timestamps + absolute paths) and SHA the normalized form for deterministic baselines.

**Reference:** `tools/assert-pptxgenjs-pin.js`, `tools/normalize-pptx-sha.js`, `tools/lint-pptxgenjs-enums.js`

### practice:tdd-with-mocked-llm: LLM-driven code is testable via dependency injection

**When this applies:** A code path's primary work is a single LLM call (extraction, classification, structured output), and you want deterministic unit tests without burning tokens or accepting flaky test runs.

Wrap the LLM client behind a `getLlm()` factory and expose a test-only injection seam (`_test_setLlm(stub)`) that swaps the factory for a synchronous fixture-returning stub. Unit tests stub the LLM entirely and assert against the surrounding logic (parsing, validation, error handling, retry). Reserve real-LLM calls for a separate E2E suite gated by an environment flag. The DI seam keeps production code unchanged; the test seam adds five lines.

**Reference:** `skills/create/scripts/lib/llm-client.js`

### practice:auto-refine-loop: Convergence loop pattern for self-improving agents

**When this applies:** You build a loop where an agent produces an artifact, critiques it, and iterates until "good enough" — and you need a robust halting strategy that avoids both premature exit and infinite spin.

Three rules together prevent the common failure modes: (1) require cycle ≥ 2 confirmation — first-cycle zero-findings is suspicious, force one more pass; (2) detect oscillation by checking whether cycle N's issue set is a subset of cycle N-2's, halt and surface the situation to the user; (3) impose a soft cap (e.g., 5 cycles) that surfaces a user choice rather than auto-stopping, plus an interrupt flag-file the user can drop to halt at the next cycle boundary. The loop owner is the agent itself, not a state machine — each "is this finding genuine?" is a judgment call.

**Reference:** `skills/create/SKILL.md` auto-refine section

### design:cookbook-variants: Ship multiple variants per slide type to defeat AI-tell uniformity

**When this applies:** You build a generative tool that produces visual outputs, and downstream readers will see many outputs side-by-side (a deck, a feed, a portfolio).

Generative tools that ship one template per output type produce homogeneous-looking results — every deck looks like every other deck, every email looks like every other email. The fix is structural: ship at least three meaningfully different variants per output type and rotate. Combine with a per-output design-DNA picker (palette × typography × motif) that explicitly diversifies against recent outputs. Validate by lint-walking the cookbook to confirm ≥3 variants exist for each registered type, and by test-rendering N decks and asserting variant histogram entropy.

**Reference:** `skills/create/references/cookbook/`

### devops:c8-100-gate: 100% coverage as a hard CI gate (no exclusions)

**When this applies:** You build a small-to-medium codebase where every line of source is reachable, and you want to prevent the slow drift toward untested branches that erodes test value over time.

`c8 --100 --check-coverage` (lines/branches/functions/statements all at 100, no per-file overrides, no excluded files) as a CI gate is brutal but durable: every new branch must come with the test that exercises it, or CI is red. The discipline forces dependency injection where it would otherwise be skipped, surfaces dead branches early, and eliminates the "we'll fix coverage later" deferral. The cost is real — verbatim third-party code must be brought under test or rewritten — but the payoff is a codebase where every line has at least one assertion behind it.

**Reference:** `.github/workflows/ci.yml` Gate 6, `package.json` test script

### design:design-dna-picker: Palette × typography × motif rotation prevents stylistic monotony

**When this applies:** A generative system produces successive outputs for the same user, and you want each output to feel distinct without making per-output choices manually.

Encode visual identity as a tuple (palette, typography, motif) and pick a fresh tuple per output via weighted rotation against the last K outputs. The picker reads a small diversity ledger (last-N choices) and avoids exact repeats; weighting accounts for compatibility (some palettes don't pair with some typographies). Surface the chosen DNA in the run's rationale doc so the user can audit and request alternatives. The pattern generalizes beyond decks — newsletters, marketing emails, social cards all benefit from the same picker.

**Reference:** `skills/create/scripts/lib/design-dna.js`
